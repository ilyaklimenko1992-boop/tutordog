# TutorDog dashboard — ролевая авторизация

- Дата: 2026-07-14
- Статус: утверждено (дизайн одобрен), готово к плану реализации
- Область: дашборд `tutordog.dxbx.ru` (`/opt/tutordog-dashboard/`, FastAPI + Starlette sessions)

## Контекст и проблема

Сейчас вход в дашборд — один общий логин (`DXBX`), сессия хранит только `{"auth": true}`. Любой вошедший видит все данные всех команд: ПДн сотрудников, итоги, детализацию ответов и банк вопросов с правильными ответами.

При этом в таблицах уже описана полноценная ролевая модель, которая никогда не подключалась к коду:

- **s3 «TutorDog Access»** (`1xjfrpZKqwOweqqNiLBfoMhCegStyBn2aCuZR5D9jhYQ`):
  - вкладка «Доступы»: `name | login | password | password_hash (bcrypt $2b$12) | role | target_audiences`;
  - вкладка «Логика ролей»: admin → всё; manager → только своя команда (фильтры ЦА и руководителя заблокированы); division_head → его ЦА целиком по всем руководителям (фильтры открыты).
- **main «Сотрудники»** (`1czMIqlFs5QtRN08EJ-DeWMxCGrwXPoAFh5kpKaZcIzA`): у каждого сотрудника `целевая_аудитория` (команда/ЦА) и `никнейм_руководителя`.

Ключ связи: `login` в «Доступах» совпадает с `никнейм_руководителя` в «Сотрудниках». Иерархия многоуровневая: division_head — руководитель менеджеров, manager — руководитель линейных сотрудников. У менеджеров подчинённых-менеджеров нет, поэтому «прямая команда» = «вся команда менеджера».

## Требования и принятые решения

1. Вход проверяется по s3 «Доступы» (bcrypt). Общий `DXBX` убирается.
2. Роль определяет видимость данных на сервере:
   - **admin** → все данные;
   - **division_head** → сотрудники с `целевая_аудитория ∈ target_audiences` (вся группа по всем руководителям);
   - **manager** → сотрудники с `никнейм_руководителя == login` (прямая команда).
3. Break-glass admin из `.env` — НЕ делаем (вход только по таблице). Единственная страховка от кратковременного сбоя Sheets — кэш списка доступов (last-known-good).
4. Пароли открытым текстом в колонке `password` — пока НЕ трогаем; проверка идёт по `password_hash`. (Долг зафиксирован в разделе «Вне области».)
5. Роль в UI не показываем.
6. Логаут — существующий стандартный `/logout`.
7. manager без команды (нет подчинённых / нет ЦА) → аккуратное уведомление вместо пустых таблиц. Текст пишет агент `wordsmith`.
8. Правильные ответы в аналитике остаются видимыми для всех ролей (это смысл аналитики срезов).

## Архитектура

**Барьер безопасности — на сервере (`main.py`).** `index.html` недоверенный: `/api/data` возвращает только разрешённые роли строки. Прятание фильтров в UI — удобство, не защита.

Разделение ответственности:
- `access.py` (новый модуль) — загрузка и кэш списка доступов из s3, проверка пароля (bcrypt), функция скоупинга.
- `main.py` — роуты, сессия, вызов скоупинга, сборка ответа `/api/data`.
- `index.html` — читает нейтральные UI-флаги из ответа, прячет ненужные фильтры, показывает уведомление о пустой команде.

### Поток аутентификации

- `GET /login` → `login.html` (визуально без изменений). Если уже авторизован → 302 `/`.
- `POST /login (username, password)`:
  1. взять список доступов (кэш, см. ниже);
  2. найти строку по `login == username`;
  3. проверить пароль `bcrypt.checkpw(password, password_hash)`;
  4. успех → записать в сессию `{auth: true, login, role, audiences}` → 303 `/`; иначе → 303 `/login?error=1`.
- `GET /logout` → `session.clear()` → `/login`.
- `GET /` → нет сессии → 302 `/login`; иначе отдать `index.html`.
- `GET /api/data` → нет сессии → 401 JSON; иначе взять полные данные (кэш 60с) → **отскоупить по роли** → вернуть `{...scoped datasets, ui:{...}}`.

### Источник доступов, кэш, отказоустойчивость

- Читаем вкладку «Доступы» тем же service account, что и остальные Sheets (`service_account.json` уже на сервере; s3 расшарен на аккаунт).
- Кэш в памяти, TTL ~5 мин, с сохранением last-known-good: при сбое обновления используем прошлый успешный снимок, чтобы кратковременный сбой Sheets не отрубал вход. Если снимка ещё нет (холодный старт при недоступном Sheets) — вход невозможен (осознанный компромисс «только таблица»).
- `target_audiences` парсим сплитом по запятой (`"MOP,TM,KVAL"`); пустое = нет ЦА.

### Сессия

Подписанная кука (текущий `SessionMiddleware`, `https_only`, `SameSite=Lax`, 30 дней): `{auth, login, role, audiences}`. PII/секретов не хранит. Роль в сессии фиксируется на момент входа; смена роли в таблице применяется после следующего входа/истечения сессии (staleness принят).

### Скоупинг данных

Из «Сотрудников» вычисляем множество разрешённых `employee_id`:
- admin → все;
- division_head → где `целевая_аудитория ∈ audiences`;
- manager → где `никнейм_руководителя == login`.

Фильтруем по этому множеству: `employees`, `sessions` (активные), `results` (итоги), `details` (детализация). `questions_bank` (содержит `правильный_ответ`) фильтруем по аудитории роли (admin — весь банк; division_head/manager — только их ЦА), чтобы менеджер чужой группы не видел вопросы и ответы другой команды.

### UI (index.html)

Сервер добавляет в ответ нейтральные флаги, **не называя роль**:
`ui: { show_audience_filter: bool, show_manager_filter: bool, empty_scope: bool }`.
- manager → оба фильтра `false` (скрыты);
- division_head → оба `true` (данные и так ограничены его ЦА);
- admin → оба `true`.

`empty_scope=true` → вместо таблиц показываем уведомление (текст от `wordsmith`). Роль не отображается. Логаут — существующая кнопка/ссылка на `/logout`.

## Зависимости, окружение, деплой

- Добавить `bcrypt` в venv и `dashboard/requirements.txt`.
- `.env`: убрать неиспользуемые `DASH_USER`/`DASH_PASS`; оставить `SESSION_SECRET`. Fail-fast переключить на проверку `SESSION_SECRET` (>=32) и наличие `service_account.json`.
- Правки `dashboard/main.py`, `dashboard/access.py`, `dashboard/index.html`, `dashboard/requirements.txt` закоммитить в репо и запушить с машины Ильи (deploy-ключ на сервере read-only).

## Тестирование (TDD)

- Юнит: функция скоупинга — admin (все), division_head (по ЦА), manager (по руководителю), пустая команда → `empty_scope`.
- Юнит: проверка пароля bcrypt (верный/неверный/несуществующий логин).
- Интеграция (`starlette TestClient`, замоканные доступы и Sheets): без сессии `/`→302, `/api/data`→401; вход неверный→`error=1`; вход верный→сессия; `/api/data` под каждой ролью возвращает только разрешённые `employee_id` и корректные `ui`-флаги.

## Вне области / принятые долги

- Колонка `password` (открытый текст) в «Доступах» остаётся; при желании удалить/ротировать — отдельной задачей (пароли уже светились в истории таблицы).
- Break-glass admin не делаем; при холодном сбое Sheets вход недоступен.
- Staleness роли до истечения/пересоздания сессии.
- Минимизация payload `/api/data` (убрать `правильный_ответ`/`pachca_user_id` из выдачи) — отдельный трек (Task Master id3), решением этой спеки правильные ответы оставлены видимыми.
